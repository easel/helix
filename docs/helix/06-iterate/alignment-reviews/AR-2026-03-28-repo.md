## Intent Summary

- **Vision**: HELIX should operate as a supervised autopilot that advances the weakest ready layer while preserving authority order, bounded actions, and tracker-first execution.
- **Requirements**: `helix run` should choose the least-power next bounded action, respect downstream artifact changes, keep execution tracker-backed, and preserve a coherent installed skill surface.
- **Features / Stories**: `FEAT-001` defines supervisory control and intervention points; `FEAT-002` defines the wrapper CLI, built-in tracker, installer, and deterministic harness.
- **Architecture / ADRs**: `ADR-001` establishes the supervisory control model; `ADR-002` establishes conservative file-backed tracker semantics with explicit write safety and concurrency visibility.
- **Technical Design**: `SD-001` and `TD-002` require bounded queue-drain routing, explicit `PLAN` and `POLISH` dispatch, queue-drift revalidation, post-implementation review handling, and coherent skill packaging.
- **Test Plans**: `TP-002` defines `bash tests/helix-cli.sh` as the deterministic proof lane for wrapper and tracker behavior.
- **Implementation Plans**: `docs/helix/04-build/implementation-plan.md` should sequence the current execution slices, but it still cites superseded predecessor issues instead of the live follow-up set.
