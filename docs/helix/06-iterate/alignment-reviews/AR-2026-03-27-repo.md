## Intent Summary

- **Vision**: HELIX should act as a supervised autopilot that advances the weakest safe layer of work while preserving authority order and bounded execution.
- **Requirements**: `helix run` should choose the least-power next bounded action, react to downstream implications of changes, preserve tracker-first execution, and tolerate concurrent local refinement safely.
- **Features / Stories**: `FEAT-001` defines supervisory control and direct intervention points; `FEAT-002` defines the wrapper CLI, tracker surface, installer, and deterministic verification contract.
- **Architecture / ADRs**: `ADR-001` selects the supervisory-control model; `ADR-002` selects a conservative file-backed tracker with explicit write-safety semantics.
- **Technical Design**: `SD-001` and `TD-002` require bounded queue-drain routing, queue-drift revalidation, machine-readable review handling, and a shared-resource skill package layout.
- **Test Plans**: `TP-002` treats `bash tests/helix-cli.sh` as the primary deterministic verification path for wrapper and tracker behavior.
- **Implementation Plans**: `docs/helix/04-build/implementation-plan.md` sequences the current execution slices and still identifies unresolved review-follow-up handling as active work.
