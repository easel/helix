## Intent Summary

- **Vision**: HELIX should act as a supervised autopilot that advances the weakest ready layer of work while preserving authority order, bounded execution, and tracker-first discipline.
- **Requirements**: `helix-run` should choose the least-power next bounded action, react to downstream implications of changed artifacts, preserve tracker-backed execution, and keep package-level skill installation coherent.
- **Features / Stories**: `FEAT-001` defines supervisory control and interactive entry points; `FEAT-002` defines the wrapper CLI, built-in tracker, installer, and deterministic verification contract.
- **Architecture / ADRs**: `ADR-001` selects the supervisory control model; `ADR-002` selects conservative file-backed tracker semantics with explicit write safety.
- **Technical Design**: `SD-001` and `TD-002` require bounded queue-drain routing, queue-drift revalidation, machine-readable review handling, and one coherent skill package surface.
- **Test Plans**: `TP-002` defines `bash tests/helix-cli.sh` as the primary deterministic proof lane for wrapper and tracker behavior.
- **Implementation Plans**: `docs/helix/04-build/implementation-plan.md` is meant to sequence the current execution slices, but it still cites superseded predecessor issues rather than the live follow-up set.
