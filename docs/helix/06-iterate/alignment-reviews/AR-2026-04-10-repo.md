## Intent Summary

- **Vision**: HELIX is a supervised autopilot that advances the weakest ready layer of the governed stack until human input is actually needed, with the tracker as the steering wheel and authority flowing downward from planning artifacts.
- **Requirements**: The PRD defines `helix run` as the primary supervisory loop, requires direct interactive entrypoints, mandates tracker-first execution, and adds plugin packaging, concerns/context digests, onboarding, testing templates, and sparse-intent intake as higher-order features.
- **Features / Stories**: FEAT-001/002 govern the CLI supervisor; FEAT-004 governs plugin packaging; FEAT-006 governs concern resolution and context-digest assembly; FEAT-007 governs the microsite and demos; FEAT-008/009/010 govern artifact/template and onboarding quality; FEAT-011 governs `helix input` and autonomy-slider intake.
- **Architecture / ADRs**: ADR-001 establishes the supervisory control model; CONTRACT-001 defines the DDx/HELIX ownership boundary; SD-001 makes the plugin/package layout part of the control model.
- **Technical Design**: TD-002 governs CLI orchestration; TD-003 adds `start`/`stop`; TD-011 governs intake/autonomy behavior.
- **Test Plans**: TP-002 is the deterministic wrapper harness for the CLI; the microsite adds Playwright e2e coverage in `website/e2e/microsite.spec.ts`.
- **Implementation Plans**: The implementation plan is now partially stale: it still treats `start`/`stop` as pending work and does not reflect the now-live `input` entrypoint or the current open queue.
