## Intent Summary

- **Vision**: HELIX is a supervised autopilot where the tracker is the steering wheel, planning and execution interleave as a double helix, and DDx owns the execution substrate below the HELIX workflow contract.
- **Requirements**: The repo must keep execution bead-first, expose a coherent public CLI and skill surface, preserve concern-aware queue metadata, and keep user-facing docs and the microsite synchronized with the numbered `docs/helix/` artifact stack.
- **Features / Stories**: FEAT-002 governs the CLI surface, FEAT-006 governs concerns/practices/context-digest propagation, FEAT-007 governs the microsite and demo reels, and FEAT-011 governs sparse-intent intake plus DDx-managed queue draining.
- **Architecture / ADRs**: CONTRACT-001 and TD-011 define the current ownership split: DDx owns `execute-bead` and `execute-loop`; HELIX owns workflow routing, prompt policy, review, and interpretation.
- **Technical Design**: Retained planning surfaces such as `helix align` are supposed to be thin bead-governed prompt launchers; queue-drain execution is supposed to delegate to DDx-managed surfaces instead of staying wrapper-owned.
- **Test Plans**: TP-002 and `tests/helix-cli.sh` are the deterministic wrapper proof lane; `tests/validate-skills.sh`, `tests/validate-state-rules.sh`, Hugo builds, and Playwright are supporting proof surfaces.
- **Implementation Plans**: The implementation plan is still a mutable queue snapshot rather than a durable contract, and it currently drifts from the live queue and public help surface.
