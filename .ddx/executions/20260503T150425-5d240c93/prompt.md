    <description>
&lt;context-digest&gt;&#xA;&lt;principles&gt;Design for Change · Design for Simplicity · Validate Your Work · Make Intent Explicit · Prefer Reversible Decisions&lt;/principles&gt;&#xA;&lt;concerns&gt;hugo-hextra | demo-asciinema&lt;/concerns&gt;&#xA;&lt;practices&gt;Theme version: Hextra v0.12.1 — pinned in website/go.mod · Hugo version: 0.159.2 extended — pinned in .github/workflows/pages.yml · Current demos: helix-quickstart (full lifecycle), helix-concerns (drift detection), helix-evolve (requirement threading) · Agent harness: ddx agent run --harness claude — not claude -p&lt;/practices&gt;&#xA;&lt;governing&gt;workflows/phases/02-design/README.md — HELIX design docs, artifact templates, and phase guidance include a first-class contract artifact with explicit selection rules and naming guidance&lt;/governing&gt;&#xA;&lt;/context-digest&gt;&#xA;&#xA;HELIX design guidance already treats contracts as a first-class artifact category (`docs/helix/02-design/contracts/`), and the design gate expects contract docs to exist, but the workflow bundle does not currently ship a dedicated contract artifact template/prompt/meta alongside solution-design, technical-design, ADR, spike, and PoC templates. At the same time, live HELIX repos are using both `API-*` and `CONTRACT-*` identifiers for documents in the contracts directory. This leaves contract-shaped specs underspecified exactly when we need them for reusable protocol/interface definitions such as OpenTelemetry telemetry capture standards, DDx/HELIX boundary contracts, execution-document conventions, CLI/library mutation surfaces, and other cross-project schemas that should be shareable without dragging along feature-level or solution-level baggage.&#xA;&#xA;Request: add an explicit first-class contract artifact type to HELIX design-phase tooling and documentation. The artifact must be suitable for normative external interface specifications, not just HTTP APIs. It should support protocol/data/interface contracts such as telemetry schemas, cross-project ownership boundaries, authoring conventions, CLI/library mutation surfaces, and other reusable contracts. The artifact definition should make clear that ADRs carry the decision/rationale (WHY), solution designs carry the feature-level approach (HOW for a feature), technical designs carry one bounded slice, and contracts carry the authoritative field-by-field/schema-level/interface-level specification that another team could implement against directly.&#xA;&#xA;Requirements from this discussion:&#xA;1. Define clear selection rules for when to use ADR vs Solution Design vs Technical Design vs Contract.&#xA;2. Ship a contract template/prompt/meta in the HELIX workflow bundle.&#xA;3. Clarify identifier and naming guidance: whether `API-*` remains a subtype, whether `CONTRACT-*` is the canonical prefix, or whether both are supported with explicit scope boundaries.&#xA;4. Make the contract artifact suitable for non-HTTP surfaces, including telemetry/export contracts.&#xA;5. Require contracts to carry exact keys/fields, types, units, enums, precedence rules, formulas, error semantics, examples, and validation checklists where relevant.&#xA;6. Ensure the design gate/enforcer language matches the chosen artifact naming and template support.&#xA;7. Preserve the ability for contracts to be shared across projects without inheriting unrelated feature or solution baggage.&#xA;8. Document how contracts should be referenced from ADRs and solution designs so the normative schema lives in one place rather than being duplicated across artifacts.
    </description>

    </acceptance>
