    <title>helix status: single-pane operator surface with parallel-run mutex</title>
    <description>
Consolidate helix status into a single surface answering: (a) current cycle number, (b) currently-executing bead id and claim age, (c) parked beads classified by reason (dep-blocked, retry-parked with next-eligible-time, orphaned-claim), (d) recently-injected supervisory beads (review-finding, alignment-review) with triggering cycle and triggering bead, (e) stop reason of the last run. Implementation is HELIX-side: scripts/helix composes existing DDx JSON primitives (ddx bead list --json, ddx bead blocked --json, ddx agent workers --json, execute-loop --json event log tail). No new ddx subcommand.&#xA;&#xA;Parallel-run mutex: second &#39;helix run&#39; on the same project root must detect an active run (via DDX_PROJECT_ROOT-scoped advisory lock or existing .helix-run.pid) and return exit code 2 with stderr &#39;helix run: already running (pid=&lt;N&gt;, started=&lt;ts&gt;)&#39; BEFORE claiming any bead.&#xA;&#xA;Migration observability counters (written to .ddx/run-state.json or a sibling file on each helix run): ddx_delegation_success_count, ddx_delegation_failure_count with reason classes (missing_binary, too_old, malformed_json, timeout), parallel_run_denied_count, supervisory_injection_count by type.
    </description>
    <acceptance>
1) helix status --json emits a single object with required fields: current_cycle (int), running_bead (object or null), parked_beads (array with {id, reason, next_eligible, age_seconds}), injected_supervisory (array with {id, type, triggering_cycle, triggering_bead}), last_stop_reason (string); jq -e verifies none are missing. 2) helix status completes in under 200ms on a tracker with 100 open beads (measured via time; test fixture seeds the tracker). 3) tests/helix-cli.sh adds a regression where launching a second &#39;helix run&#39; while the first holds the lock exits 2 with the lock-holder message within 1 second, and does not mutate the tracker. 4) .ddx/run-state.json (or documented sibling) contains the four migration counters and increments them across a scripted run.
    </acceptance>
